Internal System Blueprint · v2.0 (internal) · 16 July 2026

SiteTwin — Appletree's Site Measurement, CAD Automation & Digital Twin System

An internal productivity tool, built exclusively for Appletree Interiors' own site engineers and designers, that replaces our current paper-sketch-and-redraw process with a measured, dimensionally-accurate digital twin of every room — the single source of truth our team uses for CAD drawings, electrical/plumbing documentation, quotations, and AI-generated design visuals that actually fit the real space. This is not a commercial product and is not sold externally.

Phase 0: fully offline, manual entry, ₹0 cost Mobile-first Internal use only — single organization Integrates with existing Manufacturing ERP (Phase 1+) Residential · Commercial · Hospitality · Office
EXECUTIVE SUMMARY

The problem, in one sentence

AI interior-design tools generate beautiful rooms from a single photo, but the room they generate is not the room the client actually has — wrong wall lengths, wrong ceiling height, furniture that won't physically fit, sockets in the wrong place. SiteTwin fixes this by making measurement the foundation, not an afterthought: every wall, opening, fixture and service point is captured with real dimensions (manual, laser, or AI-assisted-and-verified) before any drawing or render is produced, so everything downstream — floor plans, elevations, BOQs, IFC/BIM exports, and AI-generated visuals — is constrained to reality.

What it captures

Hand sketches, photos, 360° video, laser-meter readings, manual entry, voice notes — fused into one structured digital twin per room.

What it produces

Dimensioned floor plans, wall elevations, MEP layouts, schedules, BOQ/QTO, and export-ready DXF/DWG/IFC/PDF files — plus a structured dataset AI render engines can use safely.

Who it's for

Appletree Interiors' own site engineers, designers, draftspeople, and project managers — across our residential, commercial, hospitality, and office projects. Built for internal use only; not licensed or sold to other firms.

Product Vision

Positioning, target users, and how SiteTwin is different from the current wave of "AI interior design" apps.

Vision statement

Every interior project in the world should be designed, documented, and rendered against a digital twin that is at least as accurate as a laser-measured as-built survey — so that what's designed on screen is exactly what gets built on site, every time, with zero re-measurement trips.

Mission

Give every designer, architect, and site engineer a mobile-first tool that turns a 20-minute guided site walk into a validated, CAD-ready, AI-ready digital twin — eliminating the manual CAD drafting, missing-measurement callbacks, and dimensionally-wrong AI renders that plague the industry today.

Solution overview — the three-layer stack

1. CaptureGuided mobile survey — sketches, photos, video, laser/manual measurements, voice notes, all geo/room-tagged
→
2. Digital TwinAI-assisted extraction + mandatory human validation → one structured, versioned, dimensionally-consistent model per room
→
3. GenerateCAD drawings, elevations, schedules, BOQ, BIM/IFC export, and a constrained dataset for AI rendering engines

The core bet: accuracy comes from measurement discipline enforced by workflow and validation rules, not from a bigger AI model. AI accelerates capture and drafting; it never replaces ground-truth measurement, and it is never allowed to silently guess a missing value.

Software Architecture

A cloud-native, event-driven architecture: synchronous CRUD for structured data, asynchronous job queues for anything AI/CV-related, and an offline-first mobile client that never blocks on the network.

Two-stage build, not one architecture: Phase 0 ships as a single self-contained offline HTML tool — no backend, no cloud, all data typed in manually and stored on the device — matching the exact pattern already used for appletree_erp_v2_1.html and appletree_erp_offline.html. The cloud/NestJS architecture below is what Phase 1 onward layers on top, once the offline tool is validated in real use and the team wants multi-device sync, AI-assisted capture, or CAD/BIM export. Nothing below is needed to get a working, useful tool into a site engineer's hands first.

Phase 0 — offline client (no backend)

Single HTML/JS Client
Guided survey form UI · manual entry for every field (room, wall, opening, electrical, plumbing, HVAC) · client-side validation rules · client-side 2D plan/elevation drawing (SVG/Canvas) · localStorage/IndexedDB persistence · photo capture stored as local file/data-URI references
Output (client-side only)
Browser print-to-PDF for the survey report/plan (same pattern as the BOQ/quotation PDFs) · JSON export of the full digital twin per project, for backup or manual handoff between devices

This is deliberately the entire Phase 0 architecture — one HTML file, run from a laptop or tablet on site, zero servers to operate or pay for.

Database Design

PostgreSQL with PostGIS for geometry, as new Prisma models alongside the existing Manufacturing ERP schema. Every survey entity belongs to a Room, every Room to a Floor, every Floor to a Project — the same Project entity already used by the ERP's CRM/quotation modules, so a project's site survey, quotation, and execution data all sit under one record.

Core entity map

User → Project → Floor → Room → {Wall, Door, Window, Beam, Column, Niche, Switchboard, Socket, LightPoint, FanHVACPoint, TVDataPoint, PlumbingPoint, Conduit, CeilingRecord, FurnitureItem} plus cross-cutting: Photo, Video, VoiceNote, Sketch, ValidationFlag, RevisionHistory, ExportJob, Signature.

Representative schema (abbreviated DDL)

-- Extends the existing Project table (already used by CRM/quotation modules)
CREATE TABLE project (
  id UUID PRIMARY KEY,
  client_name TEXT, project_name TEXT, address TEXT,
  site_engineer_id UUID, designer_id UUID,
  status TEXT CHECK (status IN ('draft','surveying','validated','designed','executed','archived')),
  created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
);

CREATE TABLE floor (id UUID PRIMARY KEY, project_id UUID REFERENCES project(id), name TEXT, level_index INT);

CREATE TABLE room (
  id UUID PRIMARY KEY, floor_id UUID REFERENCES floor(id),
  name TEXT, room_type TEXT,
  length_mm INT, width_mm INT, height_mm INT,
  diagonal_a_mm INT, diagonal_b_mm INT,
  ffl_mm INT, ceiling_height_mm INT, false_ceiling_height_mm INT,
  polygon GEOMETRY(Polygon, 0),          -- PostGIS: derived/validated room outline
  status TEXT CHECK (status IN ('pending','in_progress','flagged','validated')),
  updated_at TIMESTAMPTZ
);

CREATE TABLE wall (
  id UUID PRIMARY KEY, room_id UUID REFERENCES room(id), wall_code TEXT,
  length_mm INT, height_mm INT, thickness_mm INT,
  beam JSONB, "column" JSONB, niche JSONB, offset_mm INT, projection_mm INT,
  finish TEXT, geometry GEOMETRY(LineString, 0)
);

CREATE TABLE opening (                    -- doors & windows share a shape, discriminated by type
  id UUID PRIMARY KEY, wall_id UUID REFERENCES wall(id), type TEXT CHECK (type IN ('door','window')),
  code TEXT, width_mm INT, height_mm INT, sill_height_mm INT, lintel_height_mm INT,
  material TEXT, swing_or_opening_direction TEXT, distance_from_corner_mm INT,
  extra JSONB                             -- door: handle/lock/threshold ; window: grill/curtain box/mesh/glass type
);

CREATE TABLE electrical_point (
  id UUID PRIMARY KEY, room_id UUID REFERENCES room(id), wall_id UUID,
  category TEXT CHECK (category IN ('switchboard','socket_6a','socket_16a','socket_usb','socket_ups','socket_appliance')),
  distance_from_left_mm INT, distance_from_right_mm INT, height_from_ffl_mm INT,
  circuit_no TEXT, is_smart BOOLEAN, modules INT, functions JSONB, photo_ref UUID
);

CREATE TABLE light_point (
  id UUID PRIMARY KEY, room_id UUID REFERENCES room(id), fixture_type TEXT,
  coordinates JSONB,                      -- {x,y,z} in mm relative to room origin
  mounting_height_mm INT, circuit_no TEXT, switchboard_id UUID REFERENCES electrical_point(id), wattage INT
);

CREATE TABLE digital_twin_object (
  id UUID PRIMARY KEY, room_id UUID REFERENCES room(id),
  category TEXT, geometry JSONB, dimensions JSONB, material TEXT, finish TEXT,
  coordinates JSONB, rotation_deg NUMERIC,
  source TEXT CHECK (source IN ('manual','laser','cv_suggested','imported')),
  confidence NUMERIC, photos UUID[], notes TEXT,
  revision INT, created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
);

CREATE TABLE validation_flag (
  id UUID PRIMARY KEY, room_id UUID REFERENCES room(id), rule_id TEXT,
  severity TEXT CHECK (severity IN ('blocking','warning','info')),
  message TEXT, related_object_ids UUID[], resolved BOOLEAN DEFAULT FALSE, resolved_by UUID, resolved_at TIMESTAMPTZ
);

Electrical/lighting/HVAC/plumbing/conduit/furniture tables follow the same pattern (room-scoped, JSONB for variable attribute sets, PostGIS geometry where positional). Full field lists per module are enumerated in §15 Digital Twin Data Model.

AI Processing Pipeline

One orchestration pattern covers sketch recognition, photo/video analysis, and OCR: ingest → infer → score confidence → human review → merge into twin.

IngestMedia uploaded/queued from mobile sync
→
PreprocessDenoise, deskew, normalize, orient
→
Parallel inferenceSketch model / photo-video model / OCR model run independently
→
Confidence scoringEvery detection tagged with a score & source
→
Human-in-the-loop reviewAccept / edit / reject in mobile or web UI
→
Merge to twinOnly human-approved data becomes authoritative

Approved/corrected results are logged as training pairs, feeding periodic model retraining — the accuracy of the recognition models compounds with usage without ever letting an unverified inference reach the twin.

12

OCR Pipeline

Image prepDeskew, denoise, contrast normalize
→
Text detectionRegion proposals (DBNet/CRAFT-class detector)
→
RecognitionHandwriting-tuned transformer OCR (printed + hand-lettered dimensions/notes)
→
Dimension parsingRegex + unit normalization (mm/cm/m/ft-in) into structured values
→
Geometry associationNearest-entity linking: is this text a room label, a dimension, or a note?
→
Confidence + reviewLow-confidence reads routed to human correction, feeding active learning

Managed cloud OCR (e.g. Google Vision / AWS Textract) is used as the day-one baseline for reliability; a self-hosted handwriting-tuned model is introduced once enough corrected labels exist to fine-tune it, reducing per-page cost at scale. A fully offline fallback (lightweight on-device OCR) supports zero-connectivity capture.

This directly enforces the "never guess missing values" requirement from the product brief.

Measurement Validation Rules

Rule IDCategoryLogicResolution
VAL-010Geometric consistencyMeasured diagonal vs. Pythagorean diagonal from length/width — flag if Δ > 1%Re-measure or confirm override with note
VAL-011Geometric consistencySum of wall lengths must enclose the room polygon within toleranceRe-check individual wall entries
VAL-012Geometric consistencyInterior angle sum = (n−2)×180° ± tolerance for an n-sided roomFlag non-square/irregular room for explicit confirmation
VAL-020CompletenessAll fields required by room type present before submission is allowedHard-block submit
VAL-030Conflict detectionDoor/window swing arcs must not overlap another opening or furniture bounding boxReposition or confirm as-is with note
VAL-031Conflict detectionSwitchboard/socket height outside 300–1400mm FFL flagged as unusualConfirm intentional (e.g. counter-height) or correct
VAL-032Conflict detectionDuplicate socket at identical wall + distance + heightMerge or correct
VAL-040Cross-referenceFalse ceiling height must be < ceiling heightCorrect entry
VAL-041Cross-referenceDoor height + threshold ≤ wall heightCorrect entry
VAL-050Never-guess policyAny CV-suggested value lacking explicit human confirmation cannot enter the twin as authoritativeRequires accept/edit action
Validation flags carry a severity: blocking (submission impossible), warning (submission allowed, downstream generation shows a caveat banner), or info. The rule engine never silently corrects a value — resolution is always a logged human action.

MVP Definition

The real MVP is Phase 0: a fully offline, manual-entry tool. Everything cloud/AI-related is deliberately deferred past it.

In scope (Phase 0)

  • Runs entirely offline as a single self-contained browser tool — no server, no login, no internet required, same pattern as appletree_erp_offline.html
  • 100% manual entry: every measurement, wall, opening, electrical/plumbing/HVAC point typed in by the site engineer (a Bluetooth laser meter can feed numbers in later, but isn't required)
  • Residential project type, guided 23-step survey checklist
  • Core validation rule set (geometric consistency + completeness + conflict detection), all computed client-side
  • 2D floor plan + wall elevation generation, drawn client-side
  • Door / window / electrical schedules
  • PDF export (browser print) and digital-twin JSON export/backup
  • Data stored locally on the device used for the survey

Explicitly out of scope (deferred to Phase 1+)

  • Multi-device sync / cloud storage — one device holds one project until Phase 1
  • Roles/permissions beyond "whoever has the device" — real RBAC arrives with the cloud phase
  • Hand-sketch CV recognition
  • Photo/video CV recognition
  • DWG/BIM/IFC export (DXF/PDF only for now)
  • AI rendering integration
  • LiDAR / depth-scan ingestion
  • Any external/client-facing self-serve access beyond the view-only portal
MVP success criterion: a site engineer completes the guided survey for a 3BHK apartment (≈10 rooms) in under one working day, entirely offline on a single device, and the tool generates a validated 2D floor plan and wall elevations within ±1% of an independent laser re-survey — with zero silently-guessed values, and zero dependency on connectivity or a backend.
